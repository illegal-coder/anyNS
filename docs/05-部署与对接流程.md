# 部署与对接流程

## 1. 部署目标

anyNS 必须提供两种一站式部署方式：

- 本机一键部署。
- Docker Compose 部署。

两种部署方式都必须能完成：

- PowerDNS Recursor 启动。
- PowerDNS Authoritative 启动。
- 插件运行时启动。
- 管理 API 启动。
- DNSLog/蜜罐转发服务启动。
- 基础健康检查。
- 样例解析验证。

## 2. 本机一键部署流程

步骤：

1. 检查操作系统、权限和依赖。
2. 检查端口占用：53/udp、53/tcp、8080、8081 等。
3. 生成 PowerDNS Recursor 配置。
4. 生成 PowerDNS Authoritative 配置。
5. 生成插件运行时配置。
6. 生成管理 API 配置。
7. 生成 DNSLog/蜜罐转发配置。
8. 写入测试区和现代 RR 样例。
9. 启动服务。
10. 执行健康检查。
11. 执行 ICANN 公网解析测试。
12. 执行 HNS 样例插件测试。
13. 输出 AdGuard 上游配置。
14. 输出蜜罐 API 配置说明。

## 3. Docker Compose 部署流程

默认服务：

- `pdns-recursor`
- `pdns-authoritative`
- `anyns-plugin-runtime`
- `anyns-admin-api`
- `anyns-log-forwarder`

可选 profile：

- `coredns-security`
- `monitoring`

独立 PostgreSQL/gpgsql 模式：

```bash
export PDNS_POSTGRES_PASSWORD='replace-with-a-secret'
export PDNS_AUTH_API_KEY='replace-with-a-secret'
export PDNS_AUTH_WEBSERVER_PASSWORD='replace-with-a-secret'
docker compose -f docker-compose.gpgsql.yml up -d --wait
```

该拓扑默认将 DNS 和 API 分别绑定到回环地址的 `5301` 和 `8085`
端口，并使用 `anyns-pgsql-data` 持久化 PostgreSQL。自动化门禁为：

```bash
bash tests/acceptance/docker-gpgsql-backup-restore.sh
```

该门禁会从空数据库初始化测试区，执行权威查询，生成逻辑备份，修改记录，
恢复备份，并再次验证原始权威答案。它证明 gpgsql 数据路径可恢复，不等同于
生产环境的完整升级或灾难恢复演练。

要求：

- 使用 `.env` 管理端口、密钥、上游 DNS、蜜罐 URL。
- 使用数据卷持久化 PowerDNS 数据和插件缓存。
- 提供 healthcheck。
- 提供日志目录。
- 支持镜像版本锁定。
- 支持升级回滚。

## 4. PowerDNS 配置要求

Recursor：

- 配置 ICANN 根提示或上游递归。
- 启用缓存。
- 启用 DNSSEC 验证。
- 启用 Lua hook 或等价扩展机制。
- 配置查询日志和审计日志。

Authoritative：

- 配置本地权威区。
- 配置测试区。
- 支持 Remote Backend 或 SQL Backend。
- 支持现代 RR 和未知 RR 类型兼容表达。

## 5. AdGuard 对接

对接方式：

1. 将 anyNS 配置为 AdGuard 上游 DNS。
2. 如启用 DoH/DoT，配置对应上游地址。
3. 保留 AdGuard 现有过滤策略。
4. 在 anyNS 侧开启 AdGuard 来源视图。
5. 验证 ICANN 域名解析。
6. 验证去中心化域名解析。
7. 验证拦截事件和日志。

验收：

- AdGuard 查询可以转发到 anyNS。
- anyNS 可以返回公网域名响应。
- anyNS 可以触发去中心化插件。
- anyNS 可以生成 DNSLog。
- AdGuard 过滤不被破坏。

## 6. 蜜罐对接

配置项：

- 蜜罐 API URL。
- API Key。
- HMAC Secret。
- 批量大小。
- 请求超时。
- 重试次数。
- 失败队列目录。

流程：

1. 配置蜜罐 API 地址和密钥。
2. 发送连通性测试事件。
3. 触发安全规则。
4. 检查 DNSLog。
5. 检查蜜罐事件。
6. 模拟 API 失败。
7. 验证失败队列和重试。

## 7. 升级与回滚

### 7.1 gpgsql 备份

以下命令要求当前目录包含 `docker-compose.gpgsql.yml`，并已通过 `.env`
提供数据库名、用户和密码。备份文件必须写入访问受控、受监控且不纳入 Git
的目录。

```bash
set -a
. ./.env
set +a
COMPOSE=(docker compose --env-file .env -f docker-compose.gpgsql.yml)
BACKUP_FILE="/path/to/protected-backups/pdns-$(date -u +%Y%m%dT%H%M%SZ).sql"

"${COMPOSE[@]}" config --quiet
"${COMPOSE[@]}" exec -T pdns-postgres \
  pg_dump --clean --if-exists \
  --username="$PDNS_POSTGRES_USER" \
  --dbname="$PDNS_POSTGRES_DB" >"$BACKUP_FILE"
test -s "$BACKUP_FILE"
```

备份完成后应记录当前发布版本和 `docker compose ... config --images` 输出，
并在隔离环境执行一次恢复验证。不要把 `.env`、备份内容或实际密钥写入发布
记录。

### 7.2 gpgsql 恢复

恢复会覆盖数据库对象。先确认备份文件属于目标环境，并安排写入停机窗口：

```bash
set -a
. ./.env
set +a
COMPOSE=(docker compose --env-file .env -f docker-compose.gpgsql.yml)
BACKUP_FILE="/path/to/protected-backups/pdns-backup.sql"

test -s "$BACKUP_FILE"
"${COMPOSE[@]}" stop pdns-authoritative
"${COMPOSE[@]}" exec -T pdns-postgres \
  psql --username="$PDNS_POSTGRES_USER" \
  --dbname="$PDNS_POSTGRES_DB" \
  --set=ON_ERROR_STOP=1 <"$BACKUP_FILE"
"${COMPOSE[@]}" start pdns-authoritative
"${COMPOSE[@]}" ps
```

恢复后必须验证容器健康、SOA/NS 和至少一个已知 A/AAAA 记录，并确认 API
返回预期 zone。自动化测试使用 `www.anyns.test A` 从 `192.0.2.99` 恢复为
`192.0.2.53` 作为可重复证据。

### 7.3 发布升级

1. 在变更单中记录当前镜像摘要和 Compose 渲染结果。
2. 执行配置、PowerDNS 数据和插件配置备份。
3. 在版本控制中更新镜像摘要；不得使用浮动标签代替已审核摘要。
4. 运行 `docker compose --env-file .env -f docker-compose.gpgsql.yml config --quiet`。
5. 在隔离环境运行 `bash tests/acceptance/docker-gpgsql-backup-restore.sh`。
6. 对涉及镜像、配置或数据库迁移的发布，运行 `bash tests/acceptance/docker-gpgsql-upgrade-rollback.sh`。
7. 在维护窗口执行 `docker compose --env-file .env -f docker-compose.gpgsql.yml up -d --wait`。
8. 验证容器健康、权威 DNS 和 PowerDNS API 后再结束维护窗口。

### 7.4 发布回滚

1. 停止写入并保留失败版本的日志和数据库快照。
2. 恢复上一已验证发布中的 Compose、配置和镜像摘要。
3. 执行 `docker compose --env-file .env -f docker-compose.gpgsql.yml up -d --wait`。
4. 仅在升级改变了数据库数据或 schema 时，按 7.2 恢复升级前逻辑备份。
5. 清理不兼容缓存，并重新验证容器健康、权威 DNS 和 PowerDNS API。

当前已有自动化证据覆盖 gpgsql 初始化、数据库备份恢复、隔离升级回滚，
以及 Docker 内 source/target 项目之间的 gpgsql 数据和 private-ca 证书存储恢复。
生产管理鉴权、TLS 入口、密钥轮换、全栈升级以及跨主机灾难恢复仍需单独演练和留存结果。

### 7.5 Docker 内灾备恢复演练

`bash tests/acceptance/docker-disaster-recovery.sh` 会创建独立 source/target
Compose projects。它把 PowerDNS gpgsql 数据导出为临时 SQL dump，并将
`/var/lib/anyns/certificates` 仅在 Docker volumes 之间归档和恢复，不把私有
根或叶私钥写入 Git、日志或构建产物。恢复后验证目标环境的权威 DNS、PowerDNS
API、证书清单、证书链和叶证书指纹。

## 8. 证书签发与隔离侧载

证书签发默认关闭。启用前必须配置 PowerDNS Authoritative API、ACME directory、
账户邮箱、TOS 接受和持久化目录。先使用 staging/test CA，确认 DNS-01、续期、
吊销和 TLSA 发布后再切换生产 CA。

隔离验证：

```bash
COMPOSE_PARALLEL_LIMIT=1 \
ANYNS_CERTIFICATE_PROJECT=anyns-decentralized-certificates-review \
bash tests/acceptance/decentralized-certificates.sh
```

脚本只绑定 `127.0.0.1:28080`、`127.0.0.1:28083` 和 `127.0.0.1:25353`，
使用独立 compose project、测试区、Pebble CA 和数据卷，不覆盖默认 compose
或生产实例。验收包括 DNSKEY/RRSIG、DS/RRSIG、真实 DNS-01、证书持久化、
TLSA、续期、吊销和失败路径；它不替代真实父区 DS 和递归验证器 trust anchor
的上线验证。

管理界面回归使用无宿主机端口映射的集成网络，并限制 Compose 并发：

```bash
COMPOSE_PARALLEL_LIMIT=1 docker compose \
  -p anyns-admin-ui-review \
  -f tests/docker/compose.dns-integration.yml \
  -f tests/docker/compose.selenium.yml \
  up --build --abort-on-container-exit \
  --exit-code-from selenium-tests selenium-tests

docker compose \
  -p anyns-admin-ui-review \
  -f tests/docker/compose.dns-integration.yml \
  -f tests/docker/compose.selenium.yml \
  down -v --remove-orphans
```

该回归覆盖桌面与移动端导航、Unicode HNS 名称、Glue 必填约束、DNS 记录增删
和证书未配置状态。测试完成后必须执行 `down -v`，避免遗留测试 Zone、网络或卷。

2026-06-15 最终合并回归使用独立 project `anyns-final-merge-gpgsql`、
`anyns-final-merge-selenium` 和 `anyns-final7-certificates`。gpgsql 初始化、
记录修改、逻辑备份和恢复通过；Selenium 2 个流程通过；Pebble 侧载的 DNSSEC、
DNS-01、SAN、`0600` 私钥、重启持久化、TLSA、续期、吊销和失败路径均通过。
测试结束后 project、网络和卷已清理，生产 Compose project 未被修改。

服务器门禁使用 Go 1.18/gccgo。`golang.org/x/net v0.33.0` 会要求
`golang.org/x/crypto v0.31.0`，而后者声明 Go 1.20，因此最终依赖保持为
`x/net v0.19.0`、`x/crypto v0.17.0`、`x/text v0.14.0`。该组合覆盖仓库实际
使用的 `x/net/idna` 和 `x/crypto/acme`，并通过全量测试和 vet。

升级前同时备份 PowerDNS backend 和 `/var/lib/anyns/certificates`。回滚时恢复
旧镜像与配置；本次没有 PowerDNS schema migration。完整运维步骤见
[09-去中心化验证与证书运维.md](./09-去中心化验证与证书运维.md)。
