# anyNS 项目需求书

## 1. 项目定位

anyNS 是一个面向传统 DNS 与去中心化命名系统的统一解析、安全审计和联动平台。系统以 PowerDNS 为核心主链路，保证 ICANN 传统公网域名正常解析，同时通过 PowerDNS 插件体系接入 HNS、PNS、BNS、ENS 等主流去中心化 NS。

CoreDNS 不承担主解析链路，仅作为低优先级可选组件，用于补充 DNS 安全防护、拦截、边缘策略和现有 CoreDNS 插件生态兼容。

## 2. 建设目标

- 提供基于 PowerDNS 的去中心化 NS 解析服务器插件平台。
- 支持 PNS、HNS、BNS、ENS 等主流去中心化命名系统，并预留扩展插件机制。
- 同时兼容 ICANN 传统公网递归解析和本地权威区解析。
- 兼容 AdGuardDNS、AdGuard Home 和其他带审计、防护能力的 DNS 系统。
- 将 DNSLog、拦截行为和异常查询通过 API/Webhook 转发到蜜罐。
- 提供 DNS 攻击行为检测、风险评分和拦截插件。
- 提供完整的一站式部署、对接、升级、回滚和验收文档。

## 3. 非目标

- 不将 CoreDNS 作为去中心化解析主承载层。
- 不将 anyNS 设计为脱离 ICANN 公网 DNS 的孤立替代根服务。
- 不把 AdGuard 作为硬依赖；AdGuard 只作为可兼容的上游/下游集成对象。
- 不在首版强制实现所有生态插件的完整链上 SDK 能力；应按优先级分阶段交付。

## 4. 总体架构

- PowerDNS Recursor：主 DNS 入口，负责 ICANN 公网递归解析、缓存、DNSSEC、策略路由、插件调度、DNSLog 生成。
- PowerDNS Authoritative：负责本地权威区、测试区、托管记录、插件输出记录和现代 RR 存储。
- 插件运行时：承载所有去中心化解析插件，统一输出 DNS RRSet、审计元数据和安全标签。
- 管理 API：提供插件启停、策略更新、缓存清理、健康检查、Prometheus metrics 和审计日志查询。
- DNSLog/蜜罐联动服务：将查询、拦截和异常行为标准化为事件，并通过 API/Webhook 投递蜜罐。
- CoreDNS 可选 profile：用于额外拦截、防护、边缘策略和兼容现有 CoreDNS 插件生态。
- BIND 9.20.x 或后续稳定版：仅作为现代 RR、zonefile 和互操作测试基线，不进入核心运行链路。

## 5. 去中心化插件范围

### P1 样板插件

- Handshake/HNS：首个插件，用于验证替代根路由、PowerDNS hook、缓存、审计、失败降级和冲突策略。

### P2 Wave 1 核心插件

- ENS。
- Namecoin `.bit`。
- Stacks BNS。
- PNS-Polkadot。
- PNS-PulseChain。
- Unstoppable Domains。

### P2 Wave 2 主流扩展

- Solana SNS。
- SPACE ID，包括 `.bnb`、`.arb`、`.eth` 相关解析能力。
- TON DNS。
- Tezos Domains。
- Aptos Names。
- SuiNS。

### P2 Wave 3 生态兼容

- Freename/FNS。
- RIF Name Service/RNS。
- FIO Handle。
- OpenAlias。
- ADA Handle。
- d.id/.bit。

## 6. 命名与记录规则

- BNS 默认指 Stacks Bitcoin Name System。
- BNB Name Service 归入 SPACE ID。
- `.bit` 必须区分 Namecoin `.bit` 与 d.id/.bit。
- `WALLET` 是 IANA DNS RR type 262，不是 `.wallet` 后缀。
- 必须支持原生 `WALLET` 和 `TYPE262` 兼容表达。
- 标准 DNS 记录优先输出：`A`、`AAAA`、`CNAME`、`TXT`、`MX`、`NS`、`SRV`、`URI`、`HTTPS`、`SVCB`、`TLSA`、`CAA`、`WALLET`。
- 无法直接映射为标准 DNS RR 的链上字段，通过 `TXT`、`URI`、`WALLET` 或扩展 HTTP API 暴露。

## 7. 兼容与集成要求

- ICANN：PowerDNS Recursor 必须支持根提示或上游递归，传统公网域名不得被去中心化插件破坏。
- AdGuardDNS/AdGuard Home：通过上游 DNS、DoH/DoT、Webhook、日志导出和策略 API 对接。
- 其他审计/防护 DNS：通过标准 DNS 转发、结构化日志、Webhook 和策略同步接口兼容。
- 蜜罐：通过通用 DNS 事件 API 对接，支持后续扩展 HFish、T-Pot、Cowrie 等适配器。

## 8. 安全防护要求

系统必须支持以下检测与拦截：

- DNS tunneling。
- DGA 与高熵域名。
- NXDOMAIN flood。
- 随机子域攻击。
- DNS rebinding。
- 异常 RR 查询。
- 反射放大滥用。
- 黑白名单、sinkhole、速率限制和风险评分。
- 外部 DNSLog/OOB 回调平台域名拒绝。该能力通过显式开关启用，维护规范化平台域名列表，按 DNS 标签边界匹配平台域名及其子域；不得使用子串匹配、用户正则或不安全 wildcard。它只阻断外部回调平台查询，不关闭 anyNS 本地 DNSLog 审计存储。

拦截事件必须生成 DNSLog，并可投递蜜罐 API。投递能力必须支持 API Key/HMAC、批量、重试、幂等键、失败队列和投递状态监控。

## 9. 开发路线

- P0 基础环境：PowerDNS Recursor + Authoritative、插件运行时、管理 API、DNSLog 管线、ICANN 递归、DNSSEC、缓存、现代 RR、本机一键脚本、Docker Compose。
- P1 单一插件：实现 HNS 插件和插件模板，验证 PowerDNS hook、替代根路由、缓存、审计、失败降级。
- P2 插件并联：实现 Wave 1、Wave 2、Wave 3 插件，完成统一接口、优先级、同后缀冲突、缓存隔离、RPC/API 故障隔离。
- P3 安全防护：实现风险评分、攻击检测、拦截动作、DNSLog 蜜罐转发，并接入 CoreDNS 可选安全 profile。
- P4 文档交付：完成部署、对接、插件开发、运维监控、升级回滚和验收清单。

## 10. 验收摘要

- 可解析 ICANN 公网域名，支持 DNSSEC、缓存和 PowerDNS 权威区。
- 可查询 `WALLET/TYPE262`、`HTTPS/SVCB` 等现代 RR，并通过 BIND 9.20.x 或后续稳定版 zonefile 互操作测试。
- HNS 插件可从 PowerDNS 主链路触发，覆盖成功、无记录、超时、缓存、冲突策略和审计日志。
- 多插件并联时优先级稳定，RPC/API 故障不拖垮主解析，同后缀冲突可按策略处理。
- 可拦截指定 DNS 攻击样本，并向蜜罐 API 投递 DNSLog。
- 本机一键脚本和 Docker Compose 均可从空环境完成部署、解析、插件触发、安全拦截和事件投递。
