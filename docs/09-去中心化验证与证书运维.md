# 去中心化验证与证书运维

## 1. 术语与边界

以下机制解决不同问题，不能互相替代：

| 机制 | 被签名或证明的对象 | 验证者 | anyNS 中的职责 |
| --- | --- | --- | --- |
| DNSSEC | DNS RRSet，签名记录为 RRSIG | 递归解析器或验证客户端 | PowerDNS Authoritative 管理 DNSKEY/签名，Recursor 验证 DS -> DNSKEY -> RRSIG 链 |
| DS | 子区 DNSKEY 的摘要和算法参数 | 父区与验证解析器 | 在父区发布；anyNS 可校验 DS 或由 DNSKEY 推导 DS |
| TLSA/DANE | TLS 证书或公钥的关联数据 | 支持 DANE 的 TLS 客户端 | 存储、校验、生成和发布 TLSA；TLSA 本身不是证书 |
| ACME DNS-01 | 对 `_acme-challenge` TXT 的控制权 | ACME CA | 临时写入 TXT，完成域名控制验证，并获取 CA 签发的 X.509 证书 |
| X.509 证书签发 | CA 对证书主体、公钥和有效期签名 | TLS 客户端的 PKI 验证器 | 使用成熟 ACME 客户端；anyNS 不自建公开 CA |
| CCIP-Read | 链上合约请求链下数据，再回调链上验证 | resolver 合约 | 客户端执行 EIP-3668 获取和 callback；不绕过合约验证 |
| EIP-712 | 结构化 Web3 消息 | 钱包、合约或应用验证器 | 仅在具体合约定义了 domain/type/nonce/expiry 时适用 |

域名所有权证明不等于 DNSSEC 数据签名；DNSSEC 也不等于 TLS 证书签发。对替代根域名（例如 HNS）使用公开 WebPKI 前，必须确认 CA 能从公网 DNS 验证该名称。通常应使用 DANE、自有信任根或面向 ICANN 可验证名称的证书。

## 2. 标准、原有实现和新增实现

### 标准协议

- RFC 4033/4034/4035 定义 DNSSEC、DNSKEY、DS、RRSIG 和验证链。
- RFC 6698 与 RFC 7671 定义 TLSA/DANE 的使用、selector 和 matching type。
- RFC 8555 定义 ACME；DNS-01 使用 `_acme-challenge.<domain>` TXT。
- RFC 8659 定义 CAA。
- EIP-3668 定义 `OffchainLookup` 和 callback 流程。
- EIP-712 定义 typed structured data 签名域。

### 项目原有实现

- PowerDNS Recursor 已启用 DNSSEC 验证配置。
- HNS DNS backend 已解析 DS，Namecoin adapter 已映射 DS 和 TLSA。
- PowerDNS 管理 API/UI 的未提交工作已提供 Zone/RRSet CRUD。
- ENS/EVM adapter 原来仅支持直接 `eth_call`，未处理 `OffchainLookup`。

### 本次新增实现

- 集中式 RR 校验：A、AAAA、NS、MX、SRV、SOA、CAA、DS、DNSKEY、TLSA、SVCB、HTTPS、TXT、URI 等。
- DS 支持 SHA-1、SHA-256、SHA-384 摘要长度检查；新增 DNSKEY -> DS 推导。
- DNSKEY protocol 必须为 3，公钥必须是合法 Base64。
- TLSA 校验 usage 0-3、selector 0-1、matching type 0-2 和摘要长度。
- HNS DNS wire 支持 DNSKEY 和原生 TLSA presentation format。
- PowerDNS `cryptokeys` 管理 API/UI：读取、创建和删除 CSK/KSK/ZSK。
- ACME DNS-01 签发状态机、幂等、单作业队列、超时、有限重试、续期、吊销、TLSA 生成/发布。
- EIP-3668 CCIP-Read：sender 绑定、HTTPS 限制、私网目标拒绝、1 MiB 响应限制和递归限制。

### 明确不实现

- 不实现自有公开 CA，也不自行实现 ACME 密码学。
- 不在 PowerDNS 原生签名区手工伪造 RRSIG。
- 不实现“通用 EIP-712 DNS 证明格式”。仓库没有对应合约 type hash、chain ID、verifying contract、nonce 或 expiry 约定；擅自定义会产生互操作和重放风险。
- CCIP 网关响应的最终真实性由 resolver callback 合约验证。anyNS 不把 HTTPS 成功等同于链上签名验证。
- 当前续期为显式 API/UI 操作，没有常驻自动续期调度器。

## 3. 信任边界与威胁模型

| 风险 | 控制措施 | 剩余风险 |
| --- | --- | --- |
| DNSSEC 降级 | 使用 PowerDNS Recursor 验证模式；测试 secure/bogus/NODATA；不要把无 AD 标志当成 secure | 替代根是否有可验证 trust anchor 取决于部署 |
| DS/DNSKEY 算法混淆 | 严格字段、摘要长度和 DNSKEY protocol 校验；优先 SHA-256/SHA-384 | 算法是否被验证器支持仍由客户端决定 |
| TLSA 降级 | 仅在 DNSSEC secure 链上把 TLSA 用作 DANE 信任输入 | 普通 WebPKI 客户端可能完全忽略 TLSA |
| ACME challenge 劫持 | TXT 只写入最匹配权威区；保留同名其他 TXT；验证后清理 | PowerDNS API 凭据泄露等同于区域写权限 |
| 私钥泄露 | 账户密钥和证书私钥使用独立目录、原子写入和 `0600`；API 不返回私钥 | 宿主机 root、卷备份和容器运行时仍在信任边界内 |
| 重放 | 签发 API 强制 idempotency key，重复键绑定同一域名集合 | ACME nonce 由 ACME 客户端处理；不替代业务签名 nonce |
| 过期 | 状态记录 NotAfter，续期 API 默认只在 renew-before 窗口内允许，`force` 仅用于运维/测试 | 没有自动续期调度器，必须外部调度并监控 |
| CCIP SSRF | 只允许 HTTPS，HTTP 仅限 loopback 测试；拒绝私网、loopback、link-local 解析；每次重定向重新校验；限制响应大小 | DNS rebinding 仍需网络出口策略共同控制 |
| CCIP sender 混淆 | `OffchainLookup.sender` 必须与原始 resolver 地址相同 | callback 合约实现错误仍可能接受恶意响应 |
| EIP-712 域/nonce/expiry | 当前不接收无合约定义的 EIP-712 证明 | 后续接入时必须固定 chainId、verifyingContract、type hash、nonce 和 expiry，并维护 replay store |

日志和审计只记录作业 ID、域名、状态和错误摘要。禁止记录 ACME challenge token、账户私钥、证书私钥、PowerDNS API key 或完整 Authorization header。

## 4. 配置

```json
{
  "powerdns": {
    "authoritative_url": "http://pdns-authoritative:8081",
    "authoritative_api_key_file": "/run/secrets/pdns-api-key",
    "server_id": "localhost"
  },
  "certificates": {
    "enabled": true,
    "directory_url": "https://acme-staging-v02.api.letsencrypt.org/directory",
    "account_email": "ops@example.org",
    "accept_tos": true,
    "storage_dir": "/var/lib/anyns/certificates",
    "request_timeout": "300s",
    "dns_propagation_timeout": "120s",
    "dns_poll_interval": "2s",
    "max_attempts": 3,
    "renew_before_days": 30
  }
}
```

生产切换前先使用 CA staging directory。`accept_tos` 必须显式为 true。ACME 账户密钥由 anyNS 首次启动生成，不进入 JSON 配置或 Git。

证书管理器在 admin 进程启动时初始化。通过配置 API 修改 `certificates`、PowerDNS Authoritative URL/API key 或 `storage_dir` 后，应先等待当前作业结束，再重启 admin；这些字段不会对已构造的 ACME client 热切换。

## 5. API 使用

### DNSSEC 密钥

```http
GET /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys
POST /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys
DELETE /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys/{key_id}
POST /api/v1/powerdns/authoritative/zones/{zone}/derive-ds
```

创建 CSK：

```json
{"keytype":"csk","active":true,"published":true}
```

密钥轮换应先创建并发布新 key，等待 DNS TTL 和父区 DS 更新传播，再删除旧 key。不要先删除仍被父区 DS 引用的 KSK/CSK。

### 证书

```http
GET  /api/v1/certificates/orders
POST /api/v1/certificates/orders
GET  /api/v1/certificates/orders/{id}
GET  /api/v1/certificates/orders/{id}/certificate
POST /api/v1/certificates/orders/{id}/renew
POST /api/v1/certificates/orders/{id}/revoke
POST /api/v1/certificates/tlsa
```

签发：

```json
{
  "domains": ["example.org", "www.example.org"],
  "idempotency_key": "deploy-20260615-example-org"
}
```

TLSA 生成并发布：

```json
{
  "job_id": "certificate-job-id",
  "domain": "www.example.org",
  "port": 443,
  "protocol": "tcp",
  "usage": 3,
  "selector": 1,
  "matching_type": 1,
  "publish": true,
  "ttl": 300
}
```

`GET .../certificate` 只返回公开证书链。私钥位于 `storage_dir/certs/{job_id}/private-key.pem`，由消费证书的服务通过受限卷或部署系统读取。

## 6. 数据、迁移和回滚

本次没有修改 PowerDNS 数据库 schema。Zone、RRSet 和 cryptokey 仍由 PowerDNS 自身 backend/schema 管理，因此没有 anyNS SQL migration。

新增持久化数据：

- `storage_dir/account-key.pem`
- `storage_dir/state.json`
- `storage_dir/certs/{job_id}/fullchain.pem`
- `storage_dir/certs/{job_id}/private-key.pem`

升级前备份整个 `storage_dir` 和 PowerDNS backend。回滚旧镜像不会修改 PowerDNS schema；旧镜像不会读取证书状态目录，但目录可保留。若状态文件损坏，先恢复备份，不要重新生成账户密钥后继续吊销旧证书。进程重启时遗留的 `pending`、`validating` 或 `finalizing` 作业会被持久化为 `failed`，不会自动重放 ACME finalize；使用新的幂等键重新提交。

侧载测试会在签发后重启 admin 容器，确认状态和证书仍可读取，并在测试结束删除独立卷。该测试不覆盖生产备份系统。

## 8. 已知限制

- 公开 CA 不会因为 HNS/ENS 链上所有权自动签发公开可信证书。
- 侧载环境验证了权威区 DNSKEY/RRSIG 和 DS/RRSIG，但没有构建公共父区、递归验证器 trust anchor 和完整 DS -> DNSKEY -> RRSIG 链；生产上线必须在真实父区发布 DS 后验证 secure、bogus 和降级路径。
- TLSA 生成支持 matching type 0 和 1；校验支持 0、1、2。当前生成器未启用 SHA-512。
- 没有自动续期 scheduler、OCSP stapling 配置或证书部署 hook。
- PowerDNS API key 仍是高权限凭据，应使用 secret file、网络隔离和轮换。
- CCIP-Read 没有通用 EIP-712 本地验签；响应由链上 callback 合约判定。
