# 去中心化验证与证书运维

## 1. 术语与边界

以下机制解决不同问题，不能互相替代：

| 机制 | 被签名或证明的对象 | 验证者 | anyNS 中的职责 |
| --- | --- | --- | --- |
| DNSSEC | DNS RRSet，签名记录为 RRSIG | 递归解析器或验证客户端 | PowerDNS Authoritative 管理 DNSKEY/签名，Recursor 验证 DS -> DNSKEY -> RRSIG 链 |
| DS | 子区 DNSKEY 的摘要和算法参数 | 父区与验证解析器 | 在父区发布；anyNS 可校验 DS 或由 DNSKEY 推导 DS |
| TLSA/DANE | TLS 证书或公钥的关联数据 | 支持 DANE 的 TLS 客户端 | 存储、校验、生成和发布 TLSA；TLSA 本身不是证书 |
| ACME DNS-01 | 对 `_acme-challenge` TXT 的控制权 | ACME CA | 临时写入 TXT，完成域名控制验证，并获取 CA 签发的 X.509 证书 |
| X.509 证书签发 | CA 对证书主体、公钥和有效期签名 | TLS 客户端的 PKI 验证器 | 公开信任使用成熟 ACME 客户端；HNS/内网可选部署者自管的私有根 CA |
| CCIP-Read | 链上合约请求链下数据，再回调链上验证 | resolver 合约 | 客户端执行 EIP-3668 获取和 callback；不绕过合约验证 |
| EIP-712 | 结构化 Web3 消息 | 钱包、合约或应用验证器 | 仅在具体合约定义了 domain/type/nonce/expiry 时适用 |

域名所有权证明不等于 DNSSEC 数据签名；DNSSEC 也不等于 TLS 证书签发。对替代根域名（例如 HNS）使用公开 WebPKI 前，必须确认 CA 能从公网 DNS 验证该名称。通常应使用 DANE、自有信任根或面向 ICANN 可验证名称的证书。

## 2. 标准、原有实现和新增实现

### 标准协议

- RFC 4033/4034/4035 定义 DNSSEC、DNSKEY、DS、RRSIG 和验证链。
- RFC 6698 与 RFC 7671 定义 TLSA/DANE 的使用、selector 和 matching type。
- RFC 8555 定义 ACME；DNS-01 使用 `_acme-challenge.<domain>` TXT。
- RFC 8659 定义 CAA。
- EIP-3668 定义 `OffchainLookup` 和 callback 流程。
- EIP-712 定义 typed structured data 签名域。

### 项目原有实现

- PowerDNS Recursor 已启用 DNSSEC 验证配置。
- HNS DNS backend 已解析 DS，Namecoin adapter 已映射 DS 和 TLSA。
- PowerDNS 管理 API/UI 的未提交工作已提供 Zone/RRSet CRUD。
- ENS/EVM adapter 原来仅支持直接 `eth_call`，未处理 `OffchainLookup`。

### 本次新增实现

- 集中式 RR 校验：A、AAAA、NS、MX、SRV、SOA、CAA、DS、DNSKEY、TLSA、SVCB、HTTPS、TXT、URI 等。
- DS 支持 SHA-1、SHA-256、SHA-384 摘要长度检查；新增 DNSKEY -> DS 推导。
- DNSKEY protocol 必须为 3，公钥必须是合法 Base64。
- TLSA 校验 usage 0-3、selector 0-1、matching type 0-2 和摘要长度。
- HNS DNS wire 支持 DNSKEY 和原生 TLSA presentation format。
- PowerDNS `cryptokeys` 管理 API/UI：读取、创建和删除 CSK/KSK/ZSK。
- ACME DNS-01 签发状态机、幂等、单作业队列、超时、有限重试、续期、吊销、TLSA 生成/发布。
- `private-ca` issuer mode：与 ACME 显式分离，使用 Go `crypto/x509` clean-room 创建、导入、轮换或加载本地私有根 CA，签发 serverAuth 叶证书链，并提供不含 PEM/私钥的根 CA 元数据和禁用/启用控制端点。
- EIP-3668 CCIP-Read：sender 绑定、HTTPS 限制、私网目标拒绝、1 MiB 响应限制和递归限制。

### 明确不实现

- 不实现或宣称自有公开可信 CA，也不自行实现 ACME 协议密码学。
- 私有根 CA 仅作为显式 opt-in 的部署者信任模式；它不会自动被浏览器或操作系统信任。
- 不复制 `eskimo/certy` 源码或其旧版 PKI.js。需要的自签名、私有 CA、证书下载和 TLSA 产品能力使用 Go 标准库与项目代码 clean-room 重写。
- 不在 PowerDNS 原生签名区手工伪造 RRSIG。
- 不实现“通用 EIP-712 DNS 证明格式”。仓库没有对应合约 type hash、chain ID、verifying contract、nonce 或 expiry 约定；擅自定义会产生互操作和重放风险。
- CCIP 网关响应的最终真实性由 resolver callback 合约验证。anyNS 不把 HTTPS 成功等同于链上签名验证。
- 当前续期为显式 API/UI 操作，没有常驻自动续期调度器。

## 3. 信任边界与威胁模型

| 风险 | 控制措施 | 剩余风险 |
| --- | --- | --- |
| DNSSEC 降级 | 使用 PowerDNS Recursor 验证模式；测试 secure/bogus/NODATA；不要把无 AD 标志当成 secure | 替代根是否有可验证 trust anchor 取决于部署 |
| DS/DNSKEY 算法混淆 | 严格字段、摘要长度和 DNSKEY protocol 校验；优先 SHA-256/SHA-384 | 算法是否被验证器支持仍由客户端决定 |
| TLSA 降级 | 仅在 DNSSEC secure 链上把 TLSA 用作 DANE 信任输入 | 普通 WebPKI 客户端可能完全忽略 TLSA |
| ACME challenge 劫持 | TXT 只写入最匹配权威区；保留同名其他 TXT；验证后清理 | PowerDNS API 凭据泄露等同于区域写权限 |
| 私钥泄露 | 账户密钥和证书私钥使用独立目录、原子写入和 `0600`；API 不返回私钥 | 宿主机 root、卷备份和容器运行时仍在信任边界内 |
| 重放 | 签发 API 强制 idempotency key，重复键绑定同一域名集合 | ACME nonce 由 ACME 客户端处理；不替代业务签名 nonce |
| 过期 | 状态记录 NotAfter，续期 API 默认只在 renew-before 窗口内允许，`force` 仅用于运维/测试 | 没有自动续期调度器，必须外部调度并监控 |
| CCIP SSRF | 只允许 HTTPS，HTTP 仅限 loopback 测试；拒绝私网、loopback、link-local 解析；每次重定向重新校验；限制响应大小 | DNS rebinding 仍需网络出口策略共同控制 |
| CCIP sender 混淆 | `OffchainLookup.sender` 必须与原始 resolver 地址相同 | callback 合约实现错误仍可能接受恶意响应 |
| EIP-712 域/nonce/expiry | 当前不接收无合约定义的 EIP-712 证明 | 后续接入时必须固定 chainId、verifyingContract、type hash、nonce 和 expiry，并维护 replay store |

日志和审计只记录作业 ID、域名、状态和错误摘要。禁止记录 ACME challenge token、账户私钥、证书私钥、PowerDNS API key 或完整 Authorization header。

## 4. 配置

```json
{
  "powerdns": {
    "authoritative_url": "http://pdns-authoritative:8081",
    "authoritative_api_key_file": "/run/secrets/pdns-api-key",
    "server_id": "localhost"
  },
  "certificates": {
    "enabled": true,
    "issuer_mode": "acme",
    "directory_url": "https://acme-staging-v02.api.letsencrypt.org/directory",
    "account_email": "ops@example.org",
    "accept_tos": true,
    "storage_dir": "/var/lib/anyns/certificates",
    "request_timeout": "300s",
    "dns_propagation_timeout": "120s",
    "dns_poll_interval": "2s",
    "max_attempts": 3,
    "renew_before_days": 30,
    "crl_distribution_url": "https://ca.example.org/private-ca.crl"
  }
}
```

生产切换前先使用 CA staging directory。`accept_tos` 必须显式为 true。ACME 账户密钥由 anyNS 首次启动生成，不进入 JSON 配置或 Git。

`issuer_mode` 控制签发路径：

- `acme` 使用公开 WebPKI CA 的 DNS-01 流程，仍要求 PowerDNS Authoritative API、ACME directory、账户邮箱和 TOS 配置。
- `private-ca` 不调用 ACME directory，也不产生公开可信证书；admin 启动时在 `storage_dir/private-ca` 创建或加载部署者私有根 CA。

私有根 CA 适用于 HNS、内网或受控客户端。客户端必须显式安装并信任该根证书，或使用 DANE/TLSA 等明确的私有信任策略；anyNS 不会让浏览器或操作系统自动信任该根。

证书管理器在 admin 进程启动时初始化。通过配置 API 修改 `certificates`、PowerDNS Authoritative URL/API key 或 `storage_dir` 后，应先等待当前作业结束，再重启 admin；这些字段不会对已构造的 ACME client 热切换。

## 5. API 使用

### DNSSEC 密钥

```http
GET /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys
POST /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys
DELETE /api/v1/powerdns/authoritative/zones/{zone}/cryptokeys/{key_id}
POST /api/v1/powerdns/authoritative/zones/{zone}/derive-ds
```

创建 CSK：

```json
{"keytype":"csk","active":true,"published":true}
```

密钥轮换应先创建并发布新 key，等待 DNS TTL 和父区 DS 更新传播，再删除旧 key。不要先删除仍被父区 DS 引用的 KSK/CSK。

### 证书

```http
GET  /api/v1/certificates/orders
POST /api/v1/certificates/orders
GET  /api/v1/certificates/orders/{id}
GET  /api/v1/certificates/orders/{id}/certificate
POST /api/v1/certificates/orders/{id}/renew
POST /api/v1/certificates/orders/{id}/revoke
GET  /api/v1/certificates/private-ca/crl
GET  /api/v1/certificates/private-ca/root
GET  /api/v1/certificates/private-ca/root/certificate
GET  /api/v1/certificates/private-ca/root/trust
PATCH /api/v1/certificates/private-ca/root
POST /api/v1/certificates/private-ca/root/backup-status
POST /api/v1/certificates/private-ca/root/import
POST /api/v1/certificates/private-ca/root/rotate
POST /api/v1/certificates/tlsa
```

签发：

```json
{
  "domains": ["example.org", "www.example.org"],
  "idempotency_key": "deploy-20260615-example-org"
}
```

TLSA 生成并发布：

```json
{
  "job_id": "certificate-job-id",
  "domain": "www.example.org",
  "port": 443,
  "protocol": "tcp",
  "usage": 3,
  "selector": 1,
  "matching_type": 1,
  "publish": true,
  "ttl": 300
}
```

`GET .../certificate` 只返回公开证书链。私钥位于 `storage_dir/certs/{job_id}/private-key.pem`，由消费证书的服务通过受限卷或部署系统读取。

`GET /api/v1/certificates/private-ca/root` 是普通元数据端点，只返回私有根 CA 的 subject、serial、有效期、SHA-256 指纹、SKI/AKI、KeyUsage 和根私钥文件权限状态。它不返回根证书 PEM、根私钥或叶证书私钥。

`GET /api/v1/certificates/private-ca/root/certificate` 是显式根证书下载端点，返回当前活动私有根 CA 的公开证书 PEM，供受控客户端安装私有信任。它要求 `certificates:read`，支持 `HEAD`，只返回单个 CA 证书 PEM；不返回根私钥、叶私钥、证书存储路径或备份路径。该端点不改变信任模型，客户端仍必须由操作员显式安装并信任该根，不能把它描述为公开 WebPKI 可信。

`GET /api/v1/certificates/private-ca/root/trust` 是操作员信任交接端点，返回 `private-root` 信任模型、`public_webpki:false`、显式信任要求、根证书下载 URL、认证 CRL URL、可选公开 CRL URL、根证书指纹和安装动作。它不返回根证书 PEM、私钥、证书存储路径或备份路径；根证书仍必须通过显式下载端点获取并在安装前核对 SHA-256 指纹。

`PATCH /api/v1/certificates/private-ca/root` 接受 `{"disabled": true|false}`。禁用状态写入 `private-ca` 状态文件，并在每次签发前检查；禁用时新的叶证书签发会失败，已有证书清单、证书链下载和作业状态查询不受影响。管理审计只记录 disabled 状态和根证书指纹，不记录 PEM 或私钥。

`POST /api/v1/certificates/private-ca/root/backup-status` 接受 `{"sha256_fingerprint":"..."}`，只有与当前根 CA 指纹匹配时才会写入备份证据标记。`GET /api/v1/certificates/private-ca/root` 会返回 `backup_status.status`：没有标记为 `missing`，标记匹配当前根为 `current`，标记存在但指向旧根为 `stale`。该端点不读取外部备份系统，也不返回备份路径、PEM 或私钥；它只是让部署者在完成独立备份后记录可审计的指纹与时间。

`POST /api/v1/certificates/private-ca/root/import` 接受 `{"certificate_pem":"...","private_key_pem":"..."}`。服务端只接受自签名 CA 根证书，要求 BasicConstraints、certSign、SKI/AKI 匹配、未过期、私钥与证书公钥匹配；导入时会规范化写入单个根证书 PEM 和 PKCS#8 私钥 PEM，并把根私钥权限设为 `0600`。导入后现有证书清单仍可读取，新的 private-ca 签发使用导入根；普通响应和管理审计只包含新根 SHA-256 指纹，不返回 PEM 或私钥。

`POST /api/v1/certificates/private-ca/root/rotate` 生成新的本地私有根 CA，覆盖当前活动根并重新加载 issuer；之后的新 private-ca 叶证书使用新根签发。已有证书作业和证书链仍可读取，旧根备份标记会在元数据中显示为 `stale`。普通响应和管理审计只包含新根 SHA-256 指纹，不返回 PEM、私钥或存储路径。该操作不会把新根分发到客户端 trust store，也不会撤回已分发的旧根信任。

`GET /api/v1/certificates/private-ca/crl` 返回当前活动根签名的 PEM CRL，内容来自 anyNS 已标记为 `revoked` 且由当前根签发的叶证书 serial。CRL 用于提供最小吊销分发语义；它不包含证书链 PEM、私钥或存储路径。

`GET /api/v1/certificates/orders/{id}/ocsp` 在 `private-ca` 模式下返回 DER OCSP 响应，要求 `certificates:read`，支持 `HEAD`。issued 作业返回 `good`，revoked 作业返回 `revoked` 和吊销时间；响应由当前活动私有根签名，并只适用于当前根签发的叶证书。该端点不返回 PEM、私钥、证书存储路径或备份路径；公开 WebPKI/ACME 证书的 OCSP 仍由公开 CA 负责。

`certificates.crl_distribution_url` 仅在 `private-ca` 模式下使用。设置为带路径的 HTTP(S) URL 后，新签发的 private-ca 叶证书会包含该 URL 作为 CRL Distribution Point；Admin 服务也会在该 URL 的路径上提供无需管理凭据的 PEM CRL，前提是该路径不是 `/api`、`/healthz`、`/readyz` 或 `/metrics` 等保留路径。部署者仍需把该 URL 的主机名、TLS 和反向代理指向受控 Admin 服务，或选择静态发布位置等明确的私有信任分发路径。

在 `private-ca` 模式下，返回的证书链包含叶证书和私有根证书，不包含任何私钥。当前 private CA revoke 行为会把 anyNS 作业标记为 `revoked`、持久化 `revoked_at`，并阻止该作业继续续期；CRL 可通过认证 API 或配置的公开路径读取，作业级 OCSP 可通过认证 API 读取，但尚未实现客户端 trust store 自动化或已分发客户端信任的撤回。

## 6. 数据、迁移和回滚

本次没有修改 PowerDNS 数据库 schema。Zone、RRSet 和 cryptokey 仍由 PowerDNS 自身 backend/schema 管理，因此没有 anyNS SQL migration。

新增持久化数据：

- `storage_dir/account-key.pem`
- `storage_dir/state.json`
- `storage_dir/certs/{job_id}/fullchain.pem`
- `storage_dir/certs/{job_id}/private-key.pem`
- `storage_dir/private-ca/root-key.pem`
- `storage_dir/private-ca/root-cert.pem`
- `storage_dir/private-ca/root-backup.json`

`account-key.pem` 仅由 `acme` 模式使用；`private-ca/root-key.pem` 仅由 `private-ca` 模式使用。根私钥和叶私钥必须保持 `0600`，根证书可公开分发给受控客户端用于显式信任安装。

升级前备份整个 `storage_dir` 和 PowerDNS backend。回滚旧镜像不会修改 PowerDNS schema；旧镜像不会读取证书状态目录，但目录可保留。若状态文件损坏，先恢复备份，不要重新生成账户密钥后继续吊销旧证书；私有根 CA 场景也不要在未确认旧根失效前生成新根，否则已部署客户端会继续信任旧根而无法验证新叶证书。进程重启时遗留的 `pending`、`validating` 或 `finalizing` 作业会被持久化为 `failed`，不会自动重放 ACME finalize；使用新的幂等键重新提交。

ACME 侧载测试会在签发后重启 admin 容器，确认状态和证书仍可读取，并在测试结束删除独立卷。private CA 侧载测试还会在容器内归档并恢复 `certificates` 目录，确认根/叶私钥权限、证书清单状态、`renewal_of` 链接、`revoked_at`、根 CA 元数据和叶证书指纹保持不变。并发签发回归覆盖多个签发任务全部进入 `issued` 清单、公开清单不泄露 idempotency key，并验证底层 issuer 串行执行。两类测试都不覆盖生产备份系统。

private CA gate：

```bash
COMPOSE_PARALLEL_LIMIT=1 bash tests/acceptance/private-ca-certificates.sh
```

该脚本使用隔离 Compose profile 构建 Admin 镜像并验证：

- `private-ca` 模式签发 HNS/单标签叶证书，证书链可由测试私有根验证。
- 证书下载、吊销后证书下载和续期后证书下载均不包含私钥。
- `GET /api/v1/certificates/private-ca/root` 返回 subject、serial、有效期、SHA-256 指纹、SKI/AKI、KeyUsage 和根私钥权限状态，不返回 PEM 或私钥材料；`GET /api/v1/certificates/private-ca/root/certificate` 显式下载当前活动根证书 PEM，要求 `certificates:read`，不返回任何私钥或路径；`GET /api/v1/certificates/private-ca/root/trust` 返回私有根信任交接元数据、根证书/CRL URL 和操作员动作，明确不属于公开 WebPKI。

- 强制续期产生新的叶证书指纹；已吊销作业不能继续续期。
- 重启和容器内备份恢复后，已吊销原证书、续期证书、根/叶私钥权限和证书指纹保持一致。

该脚本使用一次性 Admin/private-ca + PowerDNS/BIND 拓扑创建 `example.hns`，验证其按 `example.` 单标签 TLD 存储和解析，覆盖 apex SOA、NS、A/AAAA glue、DNSKEY、DS、RRSIG、private-ca 叶证书签发、根证书显式下载、证书链私钥非披露、disposable HTTPS origin、默认 trust store 拒绝私有根、错误主机名拒绝、TLSA `publish:true`、作业吊销和公开 CRL 路径可达。该 gate 使用 disposable Docker volume，不发布真实父区 DS，也不自动安装客户端 trust store。

private CA 单元/API 回归还覆盖：

- 私有根禁用/启用状态通过 `PATCH /api/v1/certificates/private-ca/root` 持久化；禁用时新签发失败，重新启用后恢复签发，审计事件不包含 PEM 或私钥。
- 私有根备份证据通过 `POST /api/v1/certificates/private-ca/root/backup-status` 记录当前根指纹和时间；API 回归覆盖缺失、匹配和 stale 标记，审计事件不包含 PEM、路径或私钥。
- 私有根导入通过 `POST /api/v1/certificates/private-ca/root/import` 验证 CA 约束、SKI/AKI、公私钥匹配和未过期状态；导入后重新加载 issuer，后续叶证书由导入根签发，审计事件不包含 PEM 或私钥。
- 私有根轮换通过 `POST /api/v1/certificates/private-ca/root/rotate` 生成新的本地活动根，旧备份标记变为 `stale`，后续叶证书由轮换根签发，审计事件不包含 PEM、路径或私钥。
- 私有根信任交接通过 `GET /api/v1/certificates/private-ca/root/trust` 暴露给 `certificates:read` 主体；API 回归覆盖鉴权、capability 暴露、公开 CRL URL、安装动作字段和非披露。
- 私有根 CRL 通过 `GET /api/v1/certificates/private-ca/crl` 返回当前根签名的 PEM CRL；单元/API 回归验证 revoked 叶证书 serial、CRL 签名和私钥/证书链非披露。
- 私有根 OCSP 通过 `GET /api/v1/certificates/orders/{id}/ocsp` 返回当前根签名的 DER 响应；单元/API 回归验证 issued=`good`、revoked=`revoked`、scope、HEAD、capability 暴露和非披露。
- 配置 `certificates.crl_distribution_url` 后，新 private-ca 叶证书包含 CRL Distribution Point；配置校验要求 HTTP(S) URL，Admin 服务会在非保留 URL 路径提供无需管理凭据的 PEM CRL，仍不写入或返回私钥。
- 并发提交多个签发任务时，任务全部进入 `issued` 清单、公开清单不泄露 idempotency key，底层 issuer 保持串行执行。

## 8. 已知限制

- 公开 CA 不会因为 HNS/ENS 链上所有权自动签发公开可信证书。
- 隔离 `docker-dnssec-validation.sh` 已验证运行时 trust-anchor 注入下的 secure AD 链和 bogus SERVFAIL 链；仍未覆盖公共父区 DS 发布、生产递归验证器 trust anchor、真实替代根 trust anchor 分发和跨环境降级路径。
- TLSA 生成支持 matching type 0 和 1；校验支持 0、1、2。当前生成器未启用 SHA-512。
- 没有自动续期 scheduler、OCSP stapling 配置或证书部署 hook。
- private CA 当前支持本地根创建/加载、根导入、根轮换、根元数据、根证书显式下载、根信任交接元数据、根禁用/启用、根备份状态标记、叶证书签发、可配置 CRL Distribution Point、公开证书链下载、证书清单、有效期、强制续期、作业级吊销、当前根 CRL、配置路径 CRL 发布、作业级 OCSP、TLSA 生成/发布、HNS 单标签 TLD + HTTPS 集成 demo、并发提交串行签发、证书清单 UI 摘要、重启持久化、容器内备份恢复、Docker 内灾备恢复和短程 load/soak；尚未支持生产 trust store 自动化或跨主机灾备演练。
- PowerDNS API key 仍是高权限凭据，应使用 secret file、网络隔离和轮换。
- CCIP-Read 没有通用 EIP-712 本地验签；响应由链上 callback 合约判定。
